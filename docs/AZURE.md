# Azure Blob attachment storage

FresherDesk stores ticket attachments in **Azure Blob Storage**. Metadata lives in `x_2058901_fresher_ticket_attachment`; file bytes are uploaded via the ServiceNow server using Shared Key auth. External consumers receive time-limited **read SAS URLs** on GET ticket responses (Freshdesk / S3 presigned GET equivalent).

## Architecture

```
Agent UI / inbound email
        │
        ▼
ServiceNow (x_2058901_fresher)
  • POST upload (UI session or API key)
  • Email: sys_attachment → after-insert BR → blob + metadata
  • GET ticket / download refresh → fresh read SAS per request
        │
        ▼
Azure Blob (private container)
  {ticketSysId}/{attachmentSysId}/{safeFileName}
```

## Mitsui provisioning guide

This section is written for someone new to Azure but familiar with AWS. FresherDesk needs a **private blob container** and one **storage account access key** — similar to an S3 bucket + IAM user access key used server-side (not presigned URLs generated in Azure; ServiceNow generates SAS tokens itself).

### AWS → Azure cheat sheet

| AWS concept | Azure equivalent (what FresherDesk uses) |
|-------------|------------------------------------------|
| S3 bucket | **Storage account** + **container** (containers live inside a storage account) |
| Object key (`path/to/file.pdf`) | **Blob name** (`{ticketSysId}/{attachmentSysId}/{fileName}`) |
| Bucket policy / Block Public Access | **Disable anonymous public access** on storage account + container |
| IAM user access key (AKIA… + secret) | **Storage account access key** (Key1 or Key2 — a long base64 string) |
| S3 presigned GET URL | **Blob SAS URL** (generated by ServiceNow, not Azure Portal) |
| AWS region (e.g. `us-east-1`) | **Azure region** (e.g. `Japan East`, `East US`) — pick one close to ServiceNow users |
| VPC endpoint for S3 | **Private endpoint** for blob (optional hardening) |

**Important:** You do **not** need to create Azure AD app registrations, managed identities, or SAS policies in the portal for v1. ServiceNow signs uploads and SAS URLs with the storage account key directly (like using an IAM access key in a backend app).

### What you need before starting

| Item | Notes |
|------|-------|
| Azure subscription | Mitsui corporate subscription (or sandbox for dev) |
| Portal access | [https://portal.azure.com](https://portal.azure.com) |
| RBAC role | **Contributor** on the subscription or resource group, or a custom role that can create storage accounts |
| Naming | Storage account names are **globally unique**, 3–24 chars, lowercase letters and numbers only (no hyphens). Example: `mitsuifresherdesk01` |

### Step 1 — Create a resource group (recommended)

A resource group is like a folder for related resources (similar to tagging everything `Project=FresherDesk` in AWS).

1. Portal → search **Resource groups** → **Create**.
2. **Subscription:** choose Mitsui subscription.
3. **Resource group name:** e.g. `rg-fresherdesk-prod`.
4. **Region:** same region you will use for the storage account (e.g. **Japan East**).
5. **Review + create**.

You can skip this and create the storage account in an existing Mitsui resource group if one is already assigned to the project.

### Step 2 — Create the storage account

1. Portal → search **Storage accounts** → **Create**.
2. **Basics** tab:
   - **Subscription / Resource group:** select from Step 1.
   - **Storage account name:** e.g. `mitsuifresherdesk01` (must be globally unique — portal will tell you if taken).
   - **Region:** same as resource group.
   - **Performance:** **Standard** (not Premium — Premium is for low-latency disks, not typical file attachments).
   - **Redundancy:** **Locally-redundant storage (LRS)** is fine for attachments; use **GRS/ZRS** if Mitsui policy requires geo-redundancy.
3. **Advanced** tab (security — match AWS “block all public access”):
   - **Require secure transfer for REST API operations:** **Enabled** (HTTPS only — FresherDesk uses HTTPS).
   - **Allow enabling anonymous access on individual containers:** **Disabled** (prevents accidental public containers).
   - **Enable storage account key access:** **Enabled** (FresherDesk v1 uses account keys; disabling this would break the integration).
   - **Minimum TLS version:** **TLS 1.2** or higher.
4. **Networking** tab (optional now, can tighten later):
   - For initial setup and testing: **Enable public access from all networks**.
   - For production hardening: see [Step 5 — Network restrictions](#step-5--network-restrictions-optional) below.
5. **Review + create** → wait for deployment → **Go to resource**.

**Write down:** storage account name → this becomes `x_2058901_fresher.azure.storage_account`.

### Step 3 — Create the private container

1. On the storage account page → left menu **Data storage** → **Containers**.
2. **+ Container**.
3. **Name:** `ticket-attachments` (lowercase, hyphens OK; this becomes the ServiceNow property value).
4. **Public access level:** **Private (no anonymous access)** — this is the Azure equivalent of a private S3 bucket with Block Public Access on.
5. **Create**.

Blob path layout (automatic — no portal setup):

```
ticket-attachments/
  {ticketSysId}/
    {attachmentSysId}/
      invoice.pdf
```

### Step 4 — Copy the access key

This is the secret ServiceNow uses to upload blobs and sign read SAS URLs (like an AWS `AWS_SECRET_ACCESS_KEY` paired with the account name).

1. Storage account → **Security + networking** → **Access keys**.
2. Click **Show** next to **key1** (or key2 — either works; use key2 for rotation later while key1 is still in ServiceNow).
3. Copy **Storage account name** (sanity check) and **Key** (long base64 string).

**Security handling:**

- Store the key in Mitsui’s secret store / password manager — **not** email, Slack, or git.
- Paste into ServiceNow system property `x_2058901_fresher.azure.account_key` only (see below).
- Only users with `x_2058901_fresher.admin` (or instance admin) should set properties.

**Do not** share this key with integrators or browser clients. External apps only receive short-lived **read SAS URLs** from the FresherDesk API.

### Step 5 — Network restrictions (optional)

After basic upload/download testing works, Mitsui may want to restrict which IPs can reach blob endpoints (similar to an S3 bucket policy allowing only certain CIDRs).

1. Storage account → **Security + networking** → **Networking**.
2. **Firewalls and virtual networks** → **Enabled from selected virtual networks and IP addresses**.
3. Under **IPv4 Addresses**, add ServiceNow instance **egress IP addresses**.

Finding ServiceNow egress IPs:

- **Production / dedicated instance:** ask ServiceNow account team or check instance documentation for outbound IP allowlists.
- **PDI (developer instance):** egress may change or be shared; firewall rules are harder — many teams leave public network access enabled on PDIs and restrict only production storage accounts.

If you lock the firewall before adding ServiceNow IPs, uploads will fail with HTTP errors in ServiceNow logs (`Azure Blob upload failed with status 403`).

**Private endpoint** (advanced, like S3 VPC endpoint): possible but requires ServiceNow-side network path to Azure — usually a platform/infrastructure decision, not app-team only. Skip unless Mitsui infrastructure mandates it.

### Step 6 — Verify in Azure (before ServiceNow)

Confirm the container exists and is private:

**Option A — Azure Portal**

1. **Containers** → open `ticket-attachments`.
2. Confirm **Public access level** shows **Private**.
3. Optionally upload a test blob manually (**Upload**), then delete it.

**Option B — Azure CLI** (if installed)

```bash
az login
az storage container show \
  --account-name mitsuifresherdesk01 \
  --name ticket-attachments \
  --auth-mode login
```

**Option C — Azure Storage Explorer** (free desktop app)

1. Install [Storage Explorer](https://azure.microsoft.com/products/storage/storage-explorer/).
2. Connect → **Storage account or service** → **Account name and key**.
3. Enter account name + key → expand **Blob Containers** → confirm `ticket-attachments` exists.

### Step 7 — Configure ServiceNow (Background Script)

FresherDesk reads **`sys_properties`** via `gs.getProperty()`. Properties are **not** created on deploy — run this once as **`admin`**.

1. Log in to your instance (e.g. `https://dev385836.service-now.com`).
2. Filter navigator → **Scripts - Background** (or open `sys.scripts.do`).
3. Paste the script below, replace the placeholder values with your Azure details, and click **Run script**.

```javascript
(function () {
    var storageAccount = 'mitsuifresherdesk01';      // Azure storage account name
    var container = 'ticket-attachments';            // Container name from Step 3
    var accountKey = 'PASTE_AZURE_KEY1_OR_KEY2_HERE'; // Access key from Step 4
    var sasTtlMinutes = '15';
    var enabled = 'true';                            // 'false' to disable without deleting keys

    gs.setProperty('x_2058901_fresher.azure.storage_account', storageAccount);
    gs.setProperty('x_2058901_fresher.azure.container', container);
    gs.setProperty('x_2058901_fresher.azure.account_key', accountKey);
    gs.setProperty('x_2058901_fresher.azure.sas_ttl_minutes', sasTtlMinutes);
    gs.setProperty('x_2058901_fresher.azure.enabled', enabled);

    gs.info('FresherDesk Azure properties set for account: ' + storageAccount);
    gs.info('container=' + container + ' enabled=' + enabled + ' sas_ttl_minutes=' + sasTtlMinutes);
})();
```

4. Open **System Logs → Application Logs** (or the script output panel) and confirm you see the `FresherDesk Azure properties set` info lines with no errors.
5. Optional verify — run this second script to read values back (account key is not printed):

```javascript
gs.info('storage_account=' + gs.getProperty('x_2058901_fresher.azure.storage_account', ''));
gs.info('container=' + gs.getProperty('x_2058901_fresher.azure.container', ''));
gs.info('enabled=' + gs.getProperty('x_2058901_fresher.azure.enabled', ''));
gs.info('sas_ttl_minutes=' + gs.getProperty('x_2058901_fresher.azure.sas_ttl_minutes', ''));
gs.info('account_key configured=' + (gs.getProperty('x_2058901_fresher.azure.account_key', '') ? 'yes' : 'no'));
```

No instance restart is required. Re-run the first script any time you rotate keys or change container/account names.

Deploy the FresherDesk app first if the attachment table / REST routes are not on the instance yet (`npm run deploy`).

### Step 8 — End-to-end smoke test

1. Agent workspace → open a ticket → **Upload** a small PDF.
2. If upload fails with **503**, properties are missing or `enabled=false`.
3. If upload fails with **500** and logs mention Azure status **403**, check access key and firewall rules.
4. REST GET ticket (see [Manual test checklist](#manual-test-checklist)) → attachment should include `download_url`.
5. Open `download_url` in a browser → file downloads.
6. Send a test inbound email with an attachment → confirm metadata row exists and no `sys_attachment` remains on the ticket.

### Key rotation (when Mitsui requires it)

Azure provides **two** keys so you can rotate without downtime. Re-run the Step 7 background script with the new key:

```javascript
gs.setProperty('x_2058901_fresher.azure.account_key', 'PASTE_NEW_KEY_HERE');
gs.info('FresherDesk Azure account_key updated');
```

Rotation workflow:

1. Update the script to use **key2** → run script → test upload + download.
2. In Azure Portal → **Access keys** → **Rotate key1**.
3. Later, switch the script to the rotated key1 and rotate key2.

## ServiceNow configuration

See **[Step 7 — Configure ServiceNow (Background Script)](#step-7--configure-servicenow-background-script)** for the copy-paste script.

| Property | Example | Notes |
|----------|---------|-------|
| `x_2058901_fresher.azure.storage_account` | `mitsuifresherdesk01` | Storage account name |
| `x_2058901_fresher.azure.container` | `ticket-attachments` | Private container name |
| `x_2058901_fresher.azure.account_key` | *(secret)* | Base64 account key from Azure portal |
| `x_2058901_fresher.azure.sas_ttl_minutes` | `15` | Read SAS lifetime (default 15) |
| `x_2058901_fresher.azure.enabled` | `true` | Set `false` to disable uploads and omit SAS URLs |

Until the script has been run, uploads return **503** with a clear message; GET ticket still returns attachment metadata without `download_url`.

## Ingest paths

| Source | Flow |
|--------|------|
| **Agent UI** | Workspace upload → `POST /tickets/{id}/attachments` (session + agent role) → blob + metadata (`source=agent`) |
| **Inbound email** | Email action saves to `sys_attachment` at insert → **Migrate Email Attachments to Blob** business rule uploads to blob, writes metadata (`source=email`), deletes `sys_attachment` |
| **REST API key** | Same POST route with `X-API-Key` sets `source=api` |

## Download URLs

- **GET ticket** — each attachment includes `download_url` and `download_url_expires_at` when Azure is configured (new SAS on every GET).
- **GET `/tickets/{id}/attachments/{attachmentId}/download`** — refresh an expired SAS without reloading the full ticket. Accepts API key or agent session.

SAS tokens are **read-only**, HTTPS-only, and short-lived. The account key never leaves ServiceNow.

## Security notes

- Container must remain private; do not enable static website or anonymous blob access.
- Rotate storage account keys on a schedule; update `account_key` property after rotation.
- SAS expiry limits exposure if a URL is leaked.
- Compare to Freshdesk: their attachment URLs are time-limited presigned links; FresherDesk uses Azure Blob read SAS the same way.

## Manual test checklist

Requires Azure properties configured on the instance.

| # | Test | Expected |
|---|------|----------|
| 1 | Agent UI: upload PDF on a ticket | Metadata row in `x_2058901_fresher_ticket_attachment`; GET ticket returns `download_url`; URL downloads file in browser |
| 2 | Send email with attachment to inbound mailbox | Ticket created; attachment in metadata table; no leftover `sys_attachment` on ticket |
| 3 | GET same ticket twice via REST | Different SAS query string each time; new `download_url_expires_at` |
| 4 | Wait for SAS expiry, call download refresh route | **200** with fresh URL |
| 5 | Set `azure.enabled=false` | Upload **503**; GET ticket attachments omit `download_url` |

### REST examples (after Azure configured)

Upload (API key):

```cmd
curl.exe --ssl-no-revoke -s -X POST -H "X-API-Key: fd_live_dev_test_abc123xyz" -H "Content-Type: application/json" -d "{\"file_name\":\"test.txt\",\"content_type\":\"text/plain\",\"content_base64\":\"dGVzdA==\"}" "https://dev385836.service-now.com/api/x_2058901_fresher/v1/tickets/tickets/TKT0001001/attachments"
```

Refresh download URL:

```cmd
curl.exe --ssl-no-revoke -s -H "X-API-Key: fd_live_dev_test_abc123xyz" "https://dev385836.service-now.com/api/x_2058901_fresher/v1/tickets/tickets/TKT0001001/attachments/{attachmentSysId}/download"
```

## Local development without Azure

Leave `account_key` empty or set `enabled=false`. UI upload shows an error from the REST handler; list/load still works for metadata-only rows if any exist.

Optional offline testing: [Azurite](https://learn.microsoft.com/en-us/azure/storage/common/storage-use-azurite) with a real storage account key from the Azurite well-known dev connection string (not wired by default in this app).
